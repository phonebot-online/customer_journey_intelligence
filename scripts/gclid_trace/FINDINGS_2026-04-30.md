# gclid loss diagnosis — phonebot.com.au

## Symptom
GA4 BigQuery export shows only **26% of `google/cpc` sessions retain gclid**. Of paid Google clicks, ~74% lose attribution. Translation: 3 in 4 paid landing-page sessions are misattributed in GA4.

## Investigation summary

| Hypothesis | Result | Confidence |
|---|---|---|
| Server / Cloudflare / .htaccess strips gclid | ❌ ELIMINATED | high |
| OpenCart redirect drops query string | ❌ ELIMINATED | high |
| Client-side `history.replaceState()` cleans URL | ❌ ELIMINATED | high |
| GTM tag misconfiguration (wrong trigger) | ❌ ELIMINATED | high |
| **GTM/GA4 too slow to load before users bounce** | ✅ **CONFIRMED** | high |

## Server-side: clean (eliminated 2026-04-30)

`scripts/gclid_trace/trace.ts` followed every redirect manually for 4 URL variants. All preserved `?gclid=` through Cloudflare → Apache → OpenCart.

| Test URL | Hops | gclid preserved? |
|---|---|---|
| `https://www.phonebot.com.au/?gclid=...` | 1 (200) | ✅ |
| `https://phonebot.com.au/?gclid=...` | 2 (301 → 200) | ✅ |
| `http://www.phonebot.com.au/?gclid=...` | 2 (301 → 200) | ✅ |
| `https://www.phonebot.com.au/refurbished-phones/?gclid=...` | 1 (404) | ✅ |

## Client-side: also clean — but slow (run 2026-05-01)

`scripts/gclid_trace/trace_client.ts` (new, headless Chromium via Playwright) loaded the live homepage with test click IDs, instrumented `history.pushState`/`replaceState`, captured every outbound network request, and watched GA4 cookies land. Run twice — once on `/`, once on `/refurbished-phones`.

**Both pages: URL stays intact through page load.** No `history.replaceState` calls. Final URL still contains `?gclid=...&fbclid=...&msclkid=...`. **The URL is not the problem.**

**Both pages: GA4 + Google Ads beacons eventually fire with gclid attached.** Cookies `_gcl_aw` and `_gac_UA-54230694-1` get set with the correct gclid value.

**The only meaningful asymmetry: GTM load time.**

| Page | GTM script load | GA4 beacon fires | Total time-to-attribution |
|---|---|---|---|
| `/` (homepage) | **6,957 ms** | 8,113 ms | ~8 seconds |
| `/refurbished-phones` (category) | 829 ms | 3,430 ms | ~3.5 seconds |

## Root cause

**The homepage takes ~7 seconds to load GTM.** That is, by definition, ~7 seconds where any visitor who closes the tab, hits back, or clicks a nav link does not get their gclid recorded — because the tag that records it hasn't run yet.

Mobile paid traffic (probably 60-70% of paid clicks) on average 4G connections will routinely bounce inside that 7-second window. That bounced traffic = the 74% missing from BQ.

Category pages don't have this problem — GTM there loads in ~830ms.

This also explains why earlier server-side analysis was clean and the user was confused: there is no stripping happening anywhere. The URL is perfect, the tag config is correct, the persistence cookies work. **The problem is that 74% of homepage paid visitors never wait long enough for the tag to fire.**

## Why is homepage GTM so slow?

Looking at the request timeline on `/`:
- t=0ms: HTML returns
- t=6,957ms: GTM script (`gtm.js`) requested for first time
- t=7,835ms: ProfitMetrics `bundle.js` requested

GTM is being delayed by ~7 seconds for some reason — needs investigation. Likely candidates:
1. **`<script>` for GTM is at end of `<body>`** with no `async`/`defer`, blocked by other render tasks
2. **A pre-GTM script is doing synchronous work** that holds the main thread
3. **Consent banner blocks GTM until interaction** — if dismiss/accept hasn't happened yet, GTM stays paused
4. **GTM container is being injected by another script** that is itself slow to load
5. **CDN/cache issue specifically on homepage** (homepage gets more traffic and is less cached, or hits a different path)

Compare against `/refurbished-phones` where GTM loads at 829ms — same site, same GTM container, but 8× faster. That asymmetry points at something on the homepage specifically delaying script execution.

## Fix path (in priority order)

| Step | Owner | Effort | Effect |
|---|---|---|---|
| **1. Inspect homepage HTML for GTM script position** | Internal dev | 30 min | Identifies whether GTM is blocked by other scripts |
| **2. Move GTM container snippet to top of `<head>`** with `async` | Internal dev | 1 hr | Should drop GTM load to <1s |
| **3. Audit the consent banner** — does it block GTM until accept/dismiss? | Pat / dev | 1 hr | If yes, switch to GTM Consent Mode v2 |
| **4. Server-side GTM (sGTM)** for paid landing pages | Pat | 1 day | Captures gclid even if client-side never fires |
| **5. Re-run `trace_client.ts` after each fix** | Claude | 1 min | Confirms GTM load <1s before measuring BQ |
| **6. Re-check BQ click-id capture rate after 7 days** | Claude | automated | Confirms BQ recovers from 26% → ≥80% |

The single highest-leverage fix is step 2 (move GTM to head with async). That alone should recover most of the 74% loss.

## Side findings

- **All pixels are installed and working** — verified via `probe_fbq.ts` with a 15-second wait (the original 5-second `trace_client.ts` window was too short, since these tags fire LATE):
  - Meta Pixel: `window.fbq` v2.9.310 defined, `_fbp` browser-ID cookie set, `_fbc` correctly captures the test fbclid value, beacons fire to `connect.facebook.net` + `www.facebook.com`
  - Microsoft UET: `window.uetq` defined, `bat.bing.com` beacons fire
  - TikTok pixel: `analytics.tiktok.com` beacons fire (flag if not intentional)
  - Multiple GA4 properties (`G-JYK5VVBY82`, `G-YWTTJG2CG2`, `G-7MG984PBEY`, `AW-977669962`, plus legacy `UA-54230694-1`)
- The persistence cookies `_gcl_aw` (Google Ads), `_gac_UA-54230694-1` (legacy UA), `_fbc` (Meta) DO contain the test click-IDs — once tags fire, persistence works correctly.
- **The slow-tag-fire problem affects every paid channel, not just Google.** Mobile users bouncing in 5-7 seconds lose attribution on Meta, Microsoft, TikTok, AND Google simultaneously. The single fix (speed up homepage tag loading) recovers attribution across all of them.

## What to monitor in BigQuery after the fix

```sql
SELECT
  source, medium,
  COUNT(*) AS sessions,
  ROUND(COUNTIF(gclid IS NOT NULL) / COUNT(*) * 100, 1) AS pct_gclid
FROM `bigquery-api-494711.analytics_284223207.vw_sessions`
WHERE source = 'google' AND medium = 'cpc'
  AND session_date >= DATE_SUB(CURRENT_DATE(), INTERVAL 7 DAY)
GROUP BY source, medium;
```

Target: **≥80% gclid capture** on `google/cpc`. Currently 26%. After fix, should approach the category-page baseline (~80%+).
